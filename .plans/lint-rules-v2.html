Plan: @f0rbit/lint v2 — Boundary, Purity & Test-Rigor Rules

todo

Second wave of the lint toolchain: four new custom org rules (f0rbit/require-schema-at-boundary, f0rbit/no-ambient-effects, f0rbit/no-test-mocks, f0rbit/prefer-pipe) plus two shared-config changes (consistent-type-definitions: "type", full no-console ban) and one documented convention (fire-and-forget = void operator). Ships in two releases — 0.2.0 (cheap wins) and 0.3.0 (the hard type-aware boundary rule, piloted against corpus before release) — followed by a lightweight single-PR-per-repo rollout across the five already-clean consumers (corpus, vault, pulse, ui, devpad). Direction was fully user-decided via two A/B preference rounds; no alternatives evaluated beyond the decision record below.

Root cause

Stated goal: Add four new custom rules and two config changes to @f0rbit/lint, then roll the new versions out to the five consumer repos.

Underlying need: The v1 rollout proved the pipeline (toolchain, factory, custom typed rules, per-repo recipe) but its surveys exposed the bug classes that remain prose: boundary trust — 554 no-unsafe-assignment warns in devpad, pulse's env.BUCKET as any, corpus's 8× cause as Error — raw external data flows into typed code via casts instead of Zod parses; ambient non-determinism — the corpus backend-equivalence suite needed Date-precision special-casing purely because production code reads Date.now()/new Date() ambiently instead of through an injected clock; test rigor — vault's 25 and devpad's 65 discarded-Result-in-test fixes show tests drifting from the fakes-over-mocks discipline; and Result ergonomics — manual if (!x.ok) return x; unwrap chains where pipe() is the mandated composition. Each of these is a design-philosophy principle (Zod at every edge, provider pattern, Result discipline) that wave 1 didn't yet machine-check.

Why this matters: Prose conventions rot at the rate the v1 surveys measured — hundreds of accumulated violations per repo. Every convention converted to a rule is one less thing agents must remember. Conversely, two surveyed signals are deliberately NOT rules in this wave: the devpad e2e github_id: NULL stale-fixture incident points at fixtures-from-testing.arb() adoption (a testing-substrate change, not a lint rule) and the pulse example wrong-subpath import points at a deep-import-boundaries rule (considered, deferred) — both recorded in out-of-scope so the evidence isn't lost.

Alternatives considered

No architectural alternatives evaluated — the direction was fully user-decided via two A/B preference-elicitation rounds (2026-07-05). The decision record below is the source of truth; rule-internal spec calls derived from it live under Architecture decisions.

Decision record (user A/B outcomes — fixed)

  1. f0rbit/require-schema-at-boundary — error tier, type-aware. Raw external data (JSON.parse(...), Response.json(), equivalents) must flow through a Zod parse (.parse()/.safeParse() on a ZodType-typed receiver) before member access/use; as-casting a raw parse result is flagged. D1 rows / R2 reads are phase-in candidates, not v1 sources. Escape hatch: config-scoped overrides only.
  2. f0rbit/no-ambient-effects — warn tier, syntactic. Date.now(), argless new Date(), Math.random() flagged outside designated provider/adapter files (factory option ambient_effect_files?: string[] glob allowlist). Graduation-note pattern like devpad's tiers.
  3. @typescript-eslint/consistent-type-definitions: ["error", "type"] in the shared config — autofixable. Converts many existing interfaces (corpus backend/base.ts MetadataStorage/DataStorage, provider interfaces everywhere) — mechanical burn-down in rollout.
  4. NO return-annotation rule — inference wins. Rollout DELETES corpus AGENTS.md's prose line "All public functions returning a Result should declare it explicitly".
  5. Fire-and-forget = void operator — documented convention only; no-floating-promises' default ignoreVoid: true already enforces the bare-call ban. Docs task, no rule.
  6. f0rbit/no-test-mocks — error tier, syntactic. Flag mock/spyOn/jest imports+usages from "bun:test" (and trivial jest.* global equivalents) outside an allowlist. Fakes are the sanctioned double.
  7. no-console full ban in the shared config (no warn/error allow-list) — services use structured logging; CLI packages + scripts exempt via per-repo overrides; corpus's testing/registry.ts + testing/vending/ console.warns are DOCUMENTED design → scoped override + why-comment, do NOT redesign corpus's warn UX.
  8. f0rbit/prefer-pipe — warn tier. Flag 2+ consecutive manual Result unwraps (if (!x.ok) return x;) in one function, suggesting pipe(). Reuses must-use-result's structural Result matcher. No autofix. Single unwraps untouched.

Architecture decisions

Spec calls made within the fixed user decisions. None need user input.

  • require-schema-at-boundary — boundary sources (v1): (a) JSON.parse(...) where the callee resolves via the TypeChecker to the global JSON object (a local variable shadowing JSON does not count); (b) .json() method calls on a receiver whose type is fetch-Body-like — structurally: the receiver type has json, text, and headers members. This covers lib.dom Response/Request, undici, @cloudflare/workers-types, and framework request wrappers (Hono's HonoRequest etc.) — which is intentional: those ARE raw external data. NOT sources in v1: D1/Drizzle rows (already typed from the Drizzle schema — single source of truth holds), R2 body reads, KV.get(..., "json") — phase-in candidates in out-of-scope. JSON.parse(JSON.stringify(x)) deep-clone idiom IS flagged (push to structuredClone); a carve-out is a deferred refinement.
  • require-schema-at-boundary — sanctioned sinks: a boundary expression is sanctioned when it flows (through any number of await/parenthesis/ChainExpression peels) into: (1) the sole argument of .parse() / .safeParse() / .parseAsync() / .safeParseAsync() on a receiver whose type structurally exposes callable parse AND safeParse members (ZodType-like — version-agnostic across zod 3/4, mirroring the repo's structural-matcher philosophy; custom validators exposing both are accepted by design); (2) a slot explicitly typed unknown — a const x: unknown = declaration, an argument whose declared parameter type is unknown, a return into a declared-unknown return type, or an as unknown widening cast. Everything else — member access, destructuring, as T casts (including as unknown as T: peel as unknown layers before judging the outer cast), assignment into any typed or inferred-any slot that is then used — is a violation.
  • require-schema-at-boundary — variable tracking is single-hop: when a boundary call initialises a const/let with no type annotation (inferred any), the variable becomes a raw-boundary variable via ESLint scope analysis; each reference is judged with the same sink rules (a reference flowing into a Zod parse or an unknown slot is fine; member access / cast / re-assignment to another variable flags). No transitive hops — assigning a raw variable onward is itself a violation, which keeps the analysis bounded and honest. Two messageIds: unvalidated_boundary and cast_boundary. Rule options schema: [] — the escape hatch is config-scoped files overrides only, per the exception-map convention.
  • Both config changes live in the ESLint factory layer, not oxlint. oxlint 1.72.0 has both rules available but off by default (style / restriction categories). Owning them in the factory keeps ONE exception surface (the consumer's overrides array — the exception-map convention), keeps eslint --fix as the interface→type burn-down tool, and avoids splitting per-repo exemptions across .oxlintrc.json overrides. Each gets a why-comment at the config site and a de-dupe-survival test (last-matching-config resolution must keep them at error after oxlint_dedupe appends) so a future oxlint-config category change can't silently disable them.
  • Shared matcher module: is_result_type, boolean_literal_name, has_property, nullish_flags, peel_to_call move from must-use-result.ts to src/rules/shared/result-type.ts; must-use-result and prefer-pipe both import it. Behaviour-identical extraction — the existing must-use-result suite is the regression net.
  • prefer-pipe — "consecutive" defined operationally: an unwrap guard is an IfStatement whose test is !x.ok or x.ok === false, whose consequent is a single return x; (block or bare), and where x's type passes is_result_type. Two guards in the same statement list separated by at most one statement (the declaration feeding the next guard — the const y = ...; if (!y.ok) return y; cadence) form a run; runs of length ≥ 2 report once on the first guard. No autofix, no suggestions — the message names pipe() and points at the corpus-patterns recipe. Other guard shapes (return err(x.error), throw-in-tests) are deferred refinements.
  • no-ambient-effects — scope-verified syntactic matching: flag Date.now() calls, new Date() with zero arguments, and Math.random() calls only when Date/Math resolve to globals in ESLint's scope analysis (a local shadow doesn't flag). new Date(x) with arguments untouched. The allowlist is a factory option: ambient_effect_files?: string[] added to the Zod options schema; when present and non-empty the factory appends { files, rules: { "f0rbit/no-ambient-effects": "off" } } AFTER org_rules and BEFORE caller overrides (so callers can still out-rank it). The rule itself takes no options.
  • no-test-mocks — no factory option. The user decision allowed "factory option or files-scoped override"; the standard consumer overrides mechanism already does files-scoped exemption with zero new schema surface, so no option is added. Flags: import specifiers mock, spyOn, jest from "bun:test"; call/member usage of those bindings (scope-resolved — a local function named mock doesn't flag); and global jest.fn/spyOn/mock member calls when jest is unresolved in scope (the trivial jest equivalent). Message points at in-memory fakes / the Provider pattern as the sanctioned double.
  • Tier map: error — must-use-result (existing), no-test-mocks, require-schema-at-boundary, consistent-type-definitions, no-console; warn — no-ambient-effects, prefer-pipe. Warn-tier rules carry the graduation pattern from the v1 devpad rollout: rollout tasks append a graduation item to each repo's backlog rather than leaving warns silently permanent. All org-rule wiring stays in the single org_rules block (never preset-gated).
  • Release mechanics: 0.2.0 after phase 1 (new error-tier rules = consumer-visible behaviour change; minor bump pre-1.0), 0.3.0 after phase 2. Each bump touches all five package.jsons + packages/eslint-plugin/src/index.ts meta.version, and MUST rm bun.lock && bun install in the same commit (documented 0.1.2 pack-version gotcha; the tests/workspace.test.ts packed-tarballs test is the CI gate). Tag v* on main → OIDC trusted publishing; no new packages, so no manual bootstrap needed.
  • Rollout is one PR per repo — much lighter than v1's 3-PR recipe: formatting is already owned (no normalization step), repos are already lint-green on 0.1.5, so the delta is exactly the new rules. Sequence inside the PR: bump pin → survey (record per-rule counts in the PR body) → autofix interface→type → burn down or scope-override console/mock/boundary hits → graduation notes for warn tiers. Merge per repo settings: lint = squash-only, devpad = merge-commits-only, corpus = merge-commits-only (per its AGENTS.md), vault/pulse/ui = whatever their repo settings show.
  • Decision 4 interplay guard: deleting corpus's return-annotation prose does NOT flip the factory's @typescript-eslint/no-unnecessary-type-arguments: "off" decision — existing explicit Result<T, CorpusError> annotations remain legal (design-philosophy still says annotate at boundaries; decision 4 only rejects a rule mandating them). The why-comment at the config site stays.
