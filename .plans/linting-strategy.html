Plan: @f0rbit/lint — Ecosystem Linting + Formatting

doing

Build a new monorepo @f0rbit/lint at ~/dev/lint housing all lint + format configuration for the f0rbit ecosystem, then roll it out to corpus (pilot), the devpad scaffolder, and pulse / vault / devpad / ui. Strategy is Option 2: hybrid oxlint fast gate + thin typed ESLint layer, with oxfmt as the formatter everywhere (hard requirement). The ESLint layer carries the typed rule family plus the custom type-aware must-use-result rule over the corpus Result<T,E> shape. Configs ship as factory functions parameterised per repo (naming preset, package name). This plan completes the earlier decision-stop plan that lived at ~/dev/corpus/.plans/linting-strategy.html (now a tombstone pointing here).

Root cause

Stated goal: Add automated lint + format enforcement across all repos and scaffolded packages — a monorepo holding "all my formatting rules for all languages", with oxfmt as the formatter.

Underlying need: The style conventions (snake_case naming, no-throw/Result discipline, no-classes, strict-type family, import hygiene, indentation) currently live as prose in per-repo AGENTS.md files, enforced only by AI-agent discipline and review. Drift is caught late or never. The real ask is the "let the compiler catch bugs" principle applied to style: every convention that becomes a machine-checked rule is one less thing agents must remember and one less AGENTS.md paragraph that can rot. Formatting is the same problem in its purest form — corpus's "match the file you're editing" rule exists only because no formatter owns the answer.

Why this matters: If the toolchain can't encode the highest-value conventions — must-use-Result over the homegrown Result<T,E>, no-throw across boundaries, the typed no-unsafe-* family — a linter would exist but the root problem (late/never drift detection on the conventions that actually prevent bugs) would persist. That constraint is what eliminated Biome and oxlint-only: custom type-aware rules are ESLint-exclusive. The formatter choice is orthogonal to the linter choice, which is why oxfmt can be a hard requirement without forcing Option 4.

Option 1: ESLint-only — single-tool typed linting

ESLint v9 flat config + typescript-eslint v8 with projectService, shipped as one shared config package. Plugins: eslint-plugin-functional, eslint-plugin-unicorn, eslint-plugin-import-x, eslint-plugin-sonarjs. Every matrix row maps to a maintained rule; custom typed rules land in the same package. Was the planner's original recommendation on scoring (4.8/5), when formatting was out of scope.

Real-world examples

  • Shopify (@shopify/eslint-plugin) — one npm package carrying shared flat config + custom org rules, extended by every repo; chosen because custom rules and config travel/version together. source
  • Vercel (@vercel/style-guide) — same distribution shape; ESLint chosen because typed + org-specific rules have no equivalent elsewhere. source

Pros

  • Highest prose→rule coverage; one tool, one config surface, zero alpha dependencies.
  • Custom type-aware rules (must-use-result) are a solved problem with a first-class test harness.

Cons

  • Slowest in absolute terms (typed lint spins a TS program on every run, including editor-save).
  • Does not answer the formatting requirement at all — a formatter must be added anyway, so the "one tool" advantage evaporates once oxfmt is a hard requirement.

Option 2: Hybrid — oxlint fast gate + thin typed ESLint layer + oxfmt

recommended chosen by user

The posture oxc's own docs recommend: oxlint runs the broad syntactic rule set everywhere it's fast (editor-save, first CI step), while a slimmed-down ESLint config runs only what oxlint can't — the typed rule family, eslint-plugin-functional, naming-convention, import hygiene, and the custom rules. eslint-plugin-oxlint disables the overlap on the ESLint side (same mechanism as eslint-config-prettier), applied LAST in the flat-config array. oxfmt (same vendor as oxlint) owns formatting. Identical enforcement coverage to Option 1; the second tool is bought for feedback-loop speed and vendor coherence with the mandated formatter.

Real-world examples

  • oxc project guidance — oxlint docs recommend running oxlint alongside ESLint with eslint-plugin-oxlint de-duplication; hybrid is the officially supported posture. source
  • Shopify admin — adopted oxlint as the fast gate on a very large app while keeping ESLint org-rule infrastructure; chosen for lint-time cuts without giving up custom rules. source
  • typescript-eslint trajectory — by late 2026 the ESLint layer of a hybrid shrinks toward "custom type-aware rules only", which stays ESLint-exclusive; hybrid is the smart-money end-state. source

Pros

  • Same 100% convention coverage as Option 1 — typed + custom rules stay on the ESLint side; must-use-result ships.
  • Near-instant feedback for the syntactic majority on every save; the typed ESLint pass reserved for pre-push/CI.
  • Vendor-coherent with the mandated formatter: oxlint + oxfmt share config philosophy, release cadence, and one upstream to track.
  • Tracks the ecosystem trajectory: as tsgolint matures, rules migrate from the ESLint layer to oxlint config-by-config with no architectural change.

Cons

  • Two lint tools, two config dialects, two version streams in the shared package — a permanent carrying cost. Mitigated by centralising all version pins in one umbrella package.
  • Rule-split reasoning ("which tool flags this?") is a new failure mode when debugging CI; eslint-plugin-oxlint's buildFromOxlintConfigFile makes the overlap mechanical but the de-dupe must be regenerated whenever oxlint's ruleset is bumped.
  • oxfmt is alpha — accepted risk, user's explicit choice. Mitigation: exact version pins, one place to bump, normalization commits isolated with .git-blame-ignore-revs.

Decision log: Option 2 chosen, oxfmt hard requirement decided

User decision (2026-07-04): "pick between 2 and 4 if one of the hard requirements is oxfmt" — the orchestrator picked Option 2. Rationale: the formatter choice is linter-orthogonal (oxfmt works identically in both options), so the tiebreaker is what each option forecloses. Option 4 permanently forecloses custom type-aware rules — must-use-result over the homegrown Result<T,E> is the single most bug-preventing rule available to a never-throw codebase, and it is only buildable in an ESLint layer. Option 2 keeps it, at the cost of a second lint tool whose overlap is mechanically de-duplicated.

oxfmt's alpha status is an accepted risk — the user's explicit choice. Mitigations baked into this plan: exact version pins centralised in one umbrella package, normalization commits isolated in their own PRs with .git-blame-ignore-revs, and fmt:check (not auto-fix) in CI so a formatter behaviour change can never rewrite code in CI.

Home decision: a new monorepo @f0rbit/lint at ~/dev/lint (repo already created, empty). The user's framing — "a monorepo that has all my formatting rules for all languages" — means the layout reserves room for non-JS/TS toolchains later (packages/<tool>-config convention, e.g. a future packages/rustfmt-config); initial scope is the JS/TS toolchain only.

Architecture decisions

Calls made within the chosen approach. None require user input — the one genuinely architectural decision (toolchain + home) was already made by the user and is recorded in the decision log above.

  • Five workspace packages, one consumer install. Layout follows the agreed split (oxlint-config, eslint-config, oxfmt-config, eslint-plugin) plus a fifth umbrella package @f0rbit/lint (packages/lint) that depends on all four AND carries the exact-pinned tool binaries (oxlint, oxfmt, eslint, typescript-eslint, plugins). Consumers install one dependency; bun hoists transitive bins into node_modules/.bin, so oxlint/oxfmt/eslint resolve in scripts. This makes the alpha-version pins single-sourced (principle 1): bumping oxlint/oxfmt happens in exactly one package.json in the ecosystem.
  • Plugin package name is @f0rbit/eslint-plugin (directory packages/eslint-plugin), not the suggested eslint-plugin-f0rbit — flat config imports plugins as ordinary modules so the legacy naming convention buys nothing, and staying inside the @f0rbit scope keeps npm publishing uniform. Rule namespace in configs is f0rbit (e.g. f0rbit/must-use-result).
  • Configs are factory functions. define_lint_config({ naming: "snake_case" | "camelCase", package_name?, tsconfig_root_dir, overrides? }) in @f0rbit/eslint-config returns the flat-config array. Corpus uses snake_case functions; scaffolded packages use camelCase — one definition, per-repo parameters. The oxlint + oxfmt configs are static JSON (naming lives in the ESLint layer; nothing repo-varying on the oxlint side).
  • oxlint config sharing uses an explicit relative path: consumer .oxlintrc.json is a 3-line stub — { "extends": ["./node_modules/@f0rbit/oxlint-config/oxlintrc.json"] }. Node-module specifier resolution in oxlint's extends is not guaranteed across alpha-era versions; a relative path into bun's hoisted node_modules always works. Same pattern considered for oxfmt: if the pinned oxfmt version supports config extends, use it; otherwise f0rbit-lint init copies the canonical .oxfmtrc.json and CI drift is caught by f0rbit-lint check (byte-comparison against the package's canonical copy).
  • f0rbit-lint init/check bin ships in the umbrella package (~80 LOC): init writes the consumer stubs (.oxlintrc.json, .oxfmtrc.json, eslint.config.ts factory call, package.json scripts); check verifies the copied configs haven't drifted from the canonical package copies. This is also what the scaffolder template calls, so scaffolder and manual adoption share one code path.
  • Format style (single ecosystem-wide choice): tabs, print width 120, otherwise Prettier-compatible oxfmt defaults. Tabs matches the scaffolded-package convention in pipelines-setup ("Style: … Tabs.") and the majority of corpus source. Recorded once in @f0rbit/oxfmt-config; trivial to change before any normalization commit lands, painful after — so it is fixed now.
  • De-dupe direction: ESLint yields to oxlint. The factory appends eslint-plugin-oxlint's buildFromOxlintConfigFile("./.oxlintrc.json") output LAST in the flat-config array, so any rule oxlint owns is disabled in ESLint. Regeneration rule: whenever the pinned oxlint version is bumped, eslint-plugin-oxlint must be bumped to the matching version in the same commit (its versions track oxlint's ruleset) — captured in the lint repo's AGENTS.md.
  • Lockstep versioning, tag-triggered publish. All five packages share one version number; a v* tag triggers the release workflow (bun publish per package, NPM_TOKEN repo secret — mirror the corpus/ui publish setup if one exists; no runtime secrets in this repo, so no vault loader). No changesets — five packages moving in lockstep don't need independent versioning.
  • Exceptions are config-scoped, not comment-scoped. Known allowlists (corpus: Semaphore in concurrency.ts, create_corpus().build() throw in corpus.ts, testing/ construction throws, tests/) are files-scoped overrides in the repo-local eslint.config.ts. Inline eslint-disable requires a description and --report-unused-disable-directives is on.
  • Normalization is always its own PR (exactly one squashed commit containing only formatting), and the .git-blame-ignore-revs entry referencing the post-squash main SHA lands in the next PR. Squash-merging a combined PR would fuse formatting with logic changes and make the ignore-revs SHA unrecordable. Also set git config blame.ignoreRevsFile .git-blame-ignore-revs in each repo's AGENTS.md bootstrap notes (GitHub's blame UI honours the file automatically).
  • Commit/PR title hygiene (new workflow rule, applies to every task below): commit and PR titles are plain conventional-commit descriptions of the change — NEVER containing "phase", "task", plan names, or task IDs. Those details live in PR bodies and branch names. All integration is squash-based. The new lint repo is configured squash-merge-only from day one; corpus's merge-commit-only setting is flipped to squash-only in phase 3 (its AGENTS.md line "PRs on this repo merge with merge commits only" is updated in the same phase).
  • CI gate order in consumer repos: oxlint → eslint → oxfmt --check as a single lint step after typecheck/tests. oxlint first because it fails in milliseconds; fmt runs in check mode only (CI never rewrites code).
  • Warn→error graduation: on existing repos, rules whose violations are fixed in the adoption PR ship at error; rules needing non-mechanical refactors start at warn with a graduation item appended to that repo's TODO/backlog. No rule stays at warn permanently.
  • devpad task mirroring is deferred — devpad MCP tools are not available in this session. The verification coder for phase 1 should create the devpad project + tasks via devpad_* tools if available in its session; otherwise this stays a deferred item in out-of-scope. Task IDs will be back-filled into this plan when created.
  • Page-vs-integrate gate: n/a — no UI, pages, or nav anywhere in this plan.
