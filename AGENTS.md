# lint

Monorepo housing all lint + format configuration for the f0rbit ecosystem. Five bun-workspace packages under the `@f0rbit` scope; consumers install only the umbrella `@f0rbit/lint`. See README.md for the two-layer model (oxlint fast gate → thin typed ESLint layer → oxfmt formatting).

## Layout

```
.
├── packages/
│   ├── oxlint-config/    # static oxlintrc.json — fast syntactic gate
│   ├── oxfmt-config/     # static oxfmtrc.json — tabs, width 120
│   ├── eslint-config/    # define_lint_config factory — thin typed layer
│   ├── eslint-plugin/    # custom typed org rules (f0rbit/must-use-result)
│   └── lint/             # umbrella: tool pins + f0rbit-lint bin + factory re-export
├── tests/                # root workspace-integrity tests
├── .plans/               # HTML plans (linting-strategy.html completed; all phases shipped)
└── .github/workflows/    # ci.yml (verify) + release.yml (tag-triggered publish)
```

Layout convention: `packages/<tool>-config` per toolchain — room is reserved for non-JS/TS languages later (e.g. `packages/rustfmt-config`).

## Scripts

```sh
bun run typecheck   # tsc --noEmit
bun test            # all workspace tests
bun run lint        # oxlint . && eslint .
bun run fmt         # oxfmt .
bun run fmt:check   # oxfmt --check .
bun run verify      # the CI gate — typecheck + test + lint + fmt:check
```

bun for everything — no npm/npx in scripts or workflows, with ONE sanctioned exception: the release workflow's publish job uses npm CLI for the npm Trusted Publishers (OIDC) exchange, because `bun publish` has no OIDC support (oven-sh/bun#15601) and trusted publishing requires npm >= 11.5.1 (Node >= 22.14). bun still does the packing. CI pins bun 1.3.13 (1.3.14 segfaults — known from pulse CI).

## Conventions

- **This repo dogfoods its own configs.** Root `eslint.config.ts` calls `define_lint_config({ naming: "snake_case", ... })`; root `.oxlintrc.json` extends the workspace package; root `.oxfmtrc.json` is the canonical copy. Style: snake_case functions and variables, PascalCase types, kebab-case filenames, tabs.
- **Version pins are exact and single-sourced.** All tool binaries (`oxlint`, `oxfmt`, `eslint`, `typescript-eslint`, `jiti`) are exact-pinned in `packages/lint/package.json` ONLY. Sibling packages may repeat a pin as a devDependency for their own tests. Never introduce `^`/`~` for toolchain deps.
- **`typescript` is the one dependency the umbrella never owns, and its peer range must mirror the engine underneath, not float on a bare floor.** Every consumer repo already pins its own `typescript` to build its own source, so (unlike the toolchain binaries above) it stays a `peerDependency` on all three packages that touch typed linting — `packages/eslint-config`, `packages/eslint-plugin`, and `packages/lint` (added 0.1.5, previously had no `peerDependencies` at all). The range is `>=4.8.4 <6.1.0`, copied verbatim from `typescript-eslint@8.62.1`'s own declared peer range (`@typescript-eslint/utils`, `parser`, and `rule-tester` all declare the identical range). A bare `>=5.0.0` floor silently admits TypeScript versions typescript-eslint itself doesn't support. Surfaced while investigating an unrelated devpad CI failure — devpad pins `typescript@6.0.3` itself (inside this range either way), not a lint packaging bug.
- **De-dupe regeneration rule:** whenever the pinned `oxlint` version is bumped, bump `eslint-plugin-oxlint` to the matching version in the SAME commit — its versions track oxlint's ruleset. The factory applies its de-dupe block LAST in the flat-config array; a version mismatch silently re-enables oxlint-owned rules in ESLint.
- **Factory contract:** `define_lint_config({ naming: "snake_case" | "camelCase", package_name?, tsconfig_root_dir, overrides?, module_resolution? })` returns the flat-config array. Zod schema is the source of truth for the options type. Caller `overrides` splice in before the oxlint de-dupe block. Custom org rules get a marked insertion point (filled in phase 2).
- **`naming-convention` selector specificity is a weighted bitmask, not array order** (`packages/eslint-config/src/presets.ts`). typescript-eslint sorts selectors by `modifierWeight` (modifiers OR'd with type-modifiers into one int) and picks the FIRST match; `TypeModifiers.function` (1048576) vastly outweighs `Modifiers.const` (1). Consequence: an arrow-function const (`const Card = () => ...`) resolves through the `variable`+`types:["function"]` selector, NEVER the `variable`+`modifiers:["const"]` selector, even though both structurally match. `naming_convention_selectors_for_components` widens BOTH (plus the plain `function` selector) for exactly this reason — widening only the const-modifier selector would be dead code for arrow-function components.
- **Const-variable format must derive from the preset** (0.1.4 fix) — it used to hardcode `["snake_case", "UPPER_CASE"]` regardless of `naming`, false-flagging idiomatic camelCase locals under the camelCase preset. `const_variable_formats` in `presets.ts` is now `Record<NamingPreset, ...>` like `function_formats`.
- **`.tsx` component convention carve-out**: `tsx_component_convention(preset)` in `packages/eslint-config/src/index.ts` is a `files: ["**/*.tsx"]`-scoped second `@typescript-eslint/naming-convention` block, placed AFTER the base `naming_convention(preset)` block in the returned array — flat config resolves a rule per-file from the LAST matching config, so ordering (not just `files` scoping) is load-bearing here. It widens `naming_convention_selectors_for_components` (PascalCase added to function/const-function/const selectors) so JSX/TSX components (`Button`, `Card`, ...) are legal under EITHER repo preset. Plain `.ts` is untouched — do not widen the base `ts_files`-scoped block itself.
- **`module_resolution?: "node-esm" | "bundler"`, default `"node-esm"`** (0.1.4). `"node-esm"` is byte-identical to pre-0.1.4 (`import-x/extensions": ["error", "ignorePackages"]`, no `settings` block) — this is the "preserves corpus/vault/pulse exactly" guarantee, verified by keeping that branch a straight return with no resolver settings at all. `"bundler"` sets `pattern: { ts: "never", tsx: "never", ... }` for code extensions only (json/css/etc. stay on the `ignorePackages` default — bundlers still need explicit extensions for non-code assets) AND wires `settings: { "import-x/resolver-next": [createNodeResolver({ extensions: [...] })] }`.
- **Why bundler mode needs a custom resolver, not just the rule option**: import-x's bundled default resolver only recognises `['.mjs', '.cjs', '.js', '.json', '.node']` — it CANNOT resolve an extensionless specifier to a `.ts`/`.tsx` file. Without the `resolver-next` override, `import-x/extensions` falls back to treating an unresolvable extensionless import as having no extension at all and still demands one (via the `ignorePackages`→`always`+ignorePackages transform), even under `"never"` pattern config. The fix is `createNodeResolver({ extensions: [".ts", ".tsx", ...] })` from `eslint-plugin-import-x`'s public exports — not a separate `eslint-import-resolver-typescript` dependency.
- **TypeScript silently drops a `.tsx` file from `include` if a `.ts` file with the IDENTICAL basename also matches a wildcard pattern in the same array** (discovered building the 0.1.4 tsx-carve-out fixtures) — e.g. `include: ["*.ts", "*.tsx"]` with both `component-fixture.ts` and `component-fixture.tsx` on disk silently keeps only the `.ts` one (no error, no warning; `parseJsonConfigFileContent` just omits it) — presumably to avoid an ambiguous module specifier. Verified directly against `ts.parseJsonConfigFileContent`, independent of ESLint/typescript-eslint. Fixture pairs testing a `.ts`-vs-`.tsx` carve-out MUST use distinct basenames (`packages/eslint-config/tests/fixtures/component-fixture.tsx` vs `component-plain-fixture.ts`).
- **Fixture directories are excluded from the root `tsconfig.json`** (`"exclude": ["**/tests/fixtures/**"]`, added in 0.1.4). Fixtures are lint-rule test INPUT validated via their own local `tests/fixtures/tsconfig.json` + typescript-eslint's projectService (see `packages/eslint-config/tests/fixtures/tsconfig.json`, deliberately `"moduleResolution": "Bundler"` — divergent from the repo's own `NodeNext`), not via the root package's own `tsc --noEmit`. This surfaced when a fixture needed a genuinely extensionless cross-file import (`import-fixture.ts` → `./import-target`) that the root NodeNext config would otherwise reject with TS2835.
- **Lockstep versioning, tag-triggered publish via npm Trusted Publishers (OIDC) — no token.** All packages share one version. Tag `v*` on main → release workflow runs `verify`, then the `publish` job (with `id-token: write`) `bun pm pack`s each package (bun rewrites `workspace:*` to real versions; npm does NOT understand the workspace protocol) and `npm publish`es the tarballs in dependency order (oxlint-config, oxfmt-config, eslint-plugin → eslint-config → lint). Auth is the OIDC exchange against the trusted publisher configured on npmjs.com per package: owner `f0rbit`, repo `lint`, workflow `release.yml`, no environment, allowed action "Publish". Provenance is automatic under trusted publishing (requires the `repository` field in each package.json to match the repo — all five have it, keep it that way). No `NPM_TOKEN` secret exists or is needed.
- **Commit/PR title hygiene:** titles are plain conventional-commit descriptions — NEVER containing "phase", "task", plan names, or task IDs. Those details go in bodies. This repo is squash-merge only.
- **Exceptions are config-scoped, not comment-scoped.** Repo-specific allowlists go in `overrides` as `files`-scoped blocks in the consumer's `eslint.config.ts`. Inline `eslint-disable` needs a description; unused directives are errors.
- **Org rule decisions — the why-comments at the config sites are the decision log.** Three preset rules are OFF org-wide (findings from the corpus rollout, shipped in 0.1.2):
  - `@typescript-eslint/no-unnecessary-type-arguments` OFF in the factory's typed layer (`packages/eslint-config/src/index.ts`) — the annotate-at-boundaries convention mandates explicit `Result<T, CorpusError>` return annotations even when the error arg equals the default type parameter; the strict preset flags exactly those deliberate annotations (116 in corpus).
  - `@typescript-eslint/require-await` OFF in the factory's typed layer — Provider-pattern in-memory implementations satisfy Promise-returning interfaces with awaitless async methods; the rule has no interface-implementation allowance (63 false positives in corpus).
  - `import/no-named-as-default-member` OFF in the canonical `packages/oxlint-config/oxlintrc.json` — the testing substrate re-exports fast-check's default export as a namespace (`testing.fc`), so `fc.record(...)` member access is the mandated pattern (174 false positives in corpus). De-dupe interplay: an explicit "off" drops the rule from `buildFromOxlintConfigFile`'s disable list (126 → 125 entries); that re-enables nothing because the ESLint layer never turns the counterpart on — a factory test (`drops the oxlint-off rule from the de-dupe...`) guards this.
- **Adding a custom org rule** = four touches: `packages/eslint-plugin/src/rules/<name>.ts` (built with `ESLintUtils.RuleCreator`, structural type checks via the parser-services TypeChecker), one line in `packages/eslint-plugin/src/index.ts` `rules`, the `org_rules` block in `packages/eslint-config/src/index.ts` (before caller `overrides`, before the oxlint de-dupe — never preset-gated), and a `@typescript-eslint/rule-tester` suite under `packages/eslint-plugin/tests/`. Custom docs meta fields (e.g. `requiresTypeChecking`) must be declared via `RuleCreator<RuleDocs>`'s type parameter.

## Gotchas

- oxlint errors hard on unknown rules in `oxlintrc.json` — a rule name that doesn't exist in the pinned version fails every lint run immediately. When bumping oxlint, run `bun run lint` before committing.
- **The canonical `oxlintrc.json` is JSONC** — it carries decision-log comments. oxlint parses JSONC natively and `eslint-plugin-oxlint` reads it via `jsonc-parser`; never load it with plain `JSON.parse` (the umbrella bin only does substring checks, keep it that way).
- oxfmt (0.x alpha) has NO `extends` in its config schema — consumer `.oxfmtrc.json` files are byte-copies of the canonical `packages/oxfmt-config/oxfmtrc.json`; `f0rbit-lint check` detects drift. Repo-specific format exclusions go in `.prettierignore` (oxfmt reads it), never into `.oxfmtrc.json`.
- oxlint's `extends` resolves relative paths reliably; node-specifier resolution is not guaranteed across alpha versions. Consumer stubs use `./node_modules/@f0rbit/oxlint-config/oxlintrc.json`.
- **`bunfig.toml` must pin `linker = "hoisted"`** — in this repo AND in consumers. The umbrella model relies on transitive bins (`oxlint`, `oxfmt`, `eslint`) surfacing in the consumer's `node_modules/.bin`; bun 1.3's isolated linker breaks that. Verified: a consumer with hoisted linker installing ONLY `@f0rbit/lint` gets all four bins.
- **`jiti` is pinned in the umbrella because ESLint needs it to load `eslint.config.ts`** (TS flat config). It rides along transitively so consumers don't have to know it exists. Don't remove it from `packages/lint` deps — nothing imports it, ESLint resolves it at runtime.
- **Root `eslint.config.ts` is deliberately excluded from `tsconfig.json` `include`.** The factory lints config files via `projectService.allowDefaultProject: ["eslint.config.ts"]`, and typescript-eslint errors if a file is BOTH in the project and in allowDefaultProject. Consumers get the same layout from `f0rbit-lint init` (their `include` is `src/**`).
- **Consumers need bun/node types visible to the projectService default project** (e.g. `"types": ["bun"]` in the tsconfig at `tsconfig_root_dir`) — without them, `import.meta.dirname` in the init-written `eslint.config.ts` is error-typed and the config file flags itself with `no-unsafe-assignment`. Auto-`@types` inclusion alone was NOT sufficient in verification; the explicit `types` entry was.
- **oxlint `ignorePatterns` do not propagate through `extends`** — only the root config file's `ignorePatterns` apply. That's why the init-written `.oxlintrc.json` stub carries `["**/node_modules/**", "**/dist/**", "**/coverage/**"]` itself (the canonical config also lists them, but they only matter when it is used directly via `-c`). Related: oxlint's file walker respects `.gitignore`; a fresh repo without one would otherwise lint `node_modules`.
- **The stub-based de-dupe drops 32 entries vs reading the canonical config directly** (eslint-plugin-oxlint 1.72.0's `extends` resolution expands categories differently: 94 vs 126 disable entries). Audited: 29 of the 32 are no-ops (rules the ESLint layer never enables) and 3 double-report in both tools (`no-extraneous-class`, `no-unnecessary-type-constraint`, `no-useless-constructor` — class-adjacent, and classes are banned outright). No convention-matrix rule is affected; coverage is never lost, worst case is a duplicate diagnostic. Re-audit when bumping eslint-plugin-oxlint.
- **oxfmt formats `package.json` opinionatedly (canonical key order)** — `f0rbit-lint init` runs oxfmt on `package.json` after merging scripts so init never leaves a file that fails the `fmt:check` script it just installed. Same reason the oxlint stub is a formatted literal in `bin.ts`, not `JSON.stringify` output.
- **typescript-estree's projectService is a per-process singleton — first creator wins.** `bun test` runs every suite in one process; the eslint-config behaviour tests create the service with `allowDefaultProject: ["eslint.config.ts"]`, so the rule-tester suite in `packages/eslint-plugin` calls `clearCaches()` (from `@typescript-eslint/parser`) at module load to get a service honouring its own `allowDefaultProject: ["*.ts"]`. Without it the suite passes in isolation and fails under full `bun test` with "file.ts was not found by the project service".
- **Rule-tester cases are isolated virtual files** — shared type declarations (`tests/fixtures/result-decls.ts`) are read with `readFileSync` and prepended to every case's `code`. Don't rely on fixture-project globals leaking into the virtual default-project file; they don't.
- **`@f0rbit/eslint-plugin`'s index carries the repo's one sanctioned `as unknown as` cast**: typescript-eslint rule contexts aren't structurally assignable to eslint 10 core's `RuleDefinition` (tseslint drops deprecated context members), so the plugin export is cast to `ESLint.Plugin` at that single boundary — same widening typescript-eslint ships in its own plugin export (typed as the deliberately-loose `CompatiblePlugin`). Rule implementations themselves stay cast-free; adversaries should flag any new `as` inside `src/rules/`.
- **Consumer-simulation recipe for unpublished versions**: `bun pm pack` every workspace package, then in the throwaway consumer install ONLY `@f0rbit/lint` via `file:` and map the sibling packages through `overrides` to their tarballs — the packed umbrella manifest references exact lockstep versions that don't exist on npm yet. Hoisted linker + `"types": ["bun"]` per the gotchas above.
- **A version bump is not just `package.json`**: `packages/eslint-plugin/src/index.ts` hardcodes `meta: { name, version }` (ESLint plugin metadata). Bump it with the lockstep version — the root workspace test only checks the manifests.
- **`bun.lock` caches each workspace package's OWN version at lockfile-generation time — a version-only `package.json` edit does NOT refresh it.** `bun pm pack` rewrites `workspace:*` sibling deps using this cached lockfile version, NOT by re-reading the sibling's `package.json` at pack time. The published `@f0rbit/lint@0.1.2` and `@f0rbit/eslint-config@0.1.2` tarballs shipped sibling deps pinned to `0.1.0` (the version at the last full lockfile regen) even though every `package.json` already correctly declared `workspace:*` and was at `0.1.2` — the 0.1.1 and 0.1.2 bump commits touched only `"version"` fields and never regenerated `bun.lock`. Neither plain `bun install` nor `bun install --force` refreshes the cached workspace version in place (verified — both leave it stale); only deleting `bun.lock` and reinstalling (or a change that forces re-resolution of the workspace entry) does. `bun install --frozen-lockfile` (used in CI) does NOT catch this either, since the dependency _specifier_ (`workspace:*`) never changes — only what it silently resolves to at pack time. **Every version bump MUST `rm bun.lock && bun install` in the same commit.** The regression test in `tests/workspace.test.ts` (`packed tarballs` describe block) packs the umbrella and `eslint-config` for real via `bun pm pack` and asserts the packed manifest's `@f0rbit/*` dependency versions equal the lockstep version — this is the CI gate that catches a forgotten lockfile regen; it fails against the exact shape that shipped in 0.1.2.
- **Trusted publishing cannot create a package** — npm requires the package to already exist on the registry before a trusted publisher can be configured (docs.npmjs.com/trusted-publishers; `npm trust` has the same prerequisite). First-ever publish of any new package here is a one-time manual bootstrap: `bun pm pack` in the package dir, then `npm publish <tarball> --access public` locally while logged in (`npm login`). After that, configure the trusted publisher on npmjs.com and every subsequent version flows through the tag-triggered workflow with zero credentials.
- **Run `CI=true bun test` before pushing** — bun's `it.only` getter throws when the `CI` env var is set. The `must-use-result.test.ts` rule-tester suite wraps `itOnly` in a function to defer the access; local test runs don't catch this failure because `CI` is unset. Always verify in CI mode before pushing.
